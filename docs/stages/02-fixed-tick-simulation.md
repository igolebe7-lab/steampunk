# Этап 2. Детерминированная симуляция с фиксированными тиками

## Результат

Этап создаёт техническое ядро мира, независимое от сцены Godot, кадровой частоты и системного времени. Одинаковый сценарий и одинаковая последовательность команд дают одинаковый SHA-256 после каждого тика. Это база для повторов, сохранений, воспроизводимых ошибок и будущей синхронизации.

Этапы 1 и 2 завершены. Визуальная карта этапа 1 продолжает работать самостоятельно; подключение отображения к состоянию симуляции выполняется на следующих этапах.

## Авторские данные

Данные хранятся в типизированных Godot Resource и не изменяются симуляцией:

- ресурсы: дерево, железо, уголь и вода;
- здания: перевалочный склад, котёл и паровой молот;
- сценарий `foundation`: карта 18×18, seed `240713`, три стартовых здания.

`DefinitionCatalog` проверяет обязательные поля и уникальность идентификаторов. `ScenarioLoader` валидирует весь сценарий во временных структурах и возвращает `SimulationState` только при отсутствии ошибок. Неизвестное определение, пересечение зданий или выход footprint за карту не создают частичное состояние.

## Состояние и команды

Изменяемые данные находятся в чистых `RefCounted`-моделях. `SimulationState` хранит номер завершённого тика, seed, карту, здания, занятые клетки и следующий ID сущности.

На этом этапе поддерживается одна команда — изменение приоритета здания в диапазоне `0…4`. Этого достаточно для проверки полного технического пути без преждевременного внедрения производства:

1. команда получает целевой тик и уникальный `sequence`;
2. очередь отклоняет прошедшие тики и дубликаты;
3. команды тика сортируются по `sequence`;
4. `CommandSystem` валидирует команду до изменения состояния;
5. `InvariantChecker` проверяет ID, определения, footprint, занятость и приоритеты;
6. завершённое состояние получает канонический SHA-256.

Пример технического запуска:

```gdscript
var scenario := load("res://data/scenarios/foundation.tres") as ScenarioDef
var loaded := ScenarioLoader.new().load_scenario(scenario)
var runner := SimulationRunner.new(loaded.state)

runner.enqueue(SimulationCommand.set_building_priority(1, 10, 1, 4))
var tick_hash := runner.step()
```

## Фиксированный тик и детерминизм

`SimulationRunner` исполняет дискретные шаги. Значение `DEFAULT_TICKS_PER_SECOND = 10` зафиксировано как конфигурация будущего real-time адаптера; само ядро не использует `_process`, `delta`, `Time` или `Timer`.

`StateHasher` использует версионированный формат, включает размеры и изменяемые клетки карты, однозначно кодирует идентификаторы, сортирует клетки и здания, затем вычисляет SHA-256. Сценарный тест создаёт независимые состояния, выполняет одинаковую трассу в течение пяти тиков и сравнивает хэши каждого тика с golden-трассой, рассчитанной отдельным процессом Godot. Изменение одной команды или клетки карты обязано изменить хэш.

## Проверка

Полный gate выполняется одной командой:

```bash
./scripts/check_project.sh
```

Проверка импортирует ресурсы, запускает unit-, integration- и scenario-тесты, затем выполняет smoke-запуск главной сцены. Скрипт также отклоняет сообщения `SCRIPT ERROR`, `Parse Error` и ошибки загрузки ресурсов, даже если процесс Godot вернул нулевой код.

## Границы этапа

На этапе 2 намеренно отсутствуют рабочие, маршруты, перенос грузов, запасы в зданиях, рецепты и производство. Определения зданий содержат расширяемые поля, но производственная логика начнётся только после физической логистики этапа 3. Визуальное представление не имеет права напрямую изменять `SimulationState`: оно будет отправлять команды раннеру.
